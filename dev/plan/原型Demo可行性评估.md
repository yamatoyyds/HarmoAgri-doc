# 原型 Demo 可行性评估

| 项目 | 内容 |
|---|---|
| 文档版本 | V1.0 |
| 创建日期 | 2026-03-16 |
| 评估对象 | 10.1 寸横屏智慧农业控制原型 Demo |
| 评估结论 | 可行，建议立即启动 |

## 1. 评估结论

基于当前项目状态、现有工程基础和目标范围，先开发一个“界面先行、网关与设备占位”的原型 Demo，整体判断为：**可行性高，风险可控，投入产出比高**。

结论原因如下：

- 当前代码仓库几乎没有业务包袱，只有初始化模板，适合从 UI 骨架快速起步。
- 现有 `doc/` 文档体系已经较完整，可直接指导页面结构和功能优先级，不需要再从零定义产品范围。
- 原型阶段不依赖真实硬件和联调资源，能够显著降低阻塞因素。
- 目标设备和使用方向较明确，`10.1` 寸横屏有利于采用稳定的控制台布局。
- 原型 Demo 能快速产出可视成果，有利于后续评审、答辩和团队协作。

## 2. 基线状态评估

### 2.1 当前工程状态

当前状态属于“工程已初始化，业务未开始”的阶段：

- 已具备 OpenHarmony 应用工程结构。
- 已具备 `entry` 模块、`UIAbility`、基础资源文件。
- 首页仍为模板 `Hello World`。
- 尚未形成业务页面、组件体系、mock 数据层和状态管理结构。

评估：

- 对原型开发是利好，因为可在较少历史负担下统一架构。

### 2.2 文档支撑程度

现有文档已覆盖：

- 需求范围。
- 页面规格。
- 接口规划。
- 数据表规划。
- 测试思路。
- 交付与演示路径。

评估：

- 文档支撑充足，完全可以作为原型页面信息架构输入。

## 3. 技术可行性评估

### 3.1 平台适配可行性

评估结论：高。

原因：

- 工程本身就是 OpenHarmony 应用模板。
- `tablet` 设备类型已在模块配置中声明。
- 原型阶段主要是 ArkUI 页面开发，属于平台擅长场景。
- 横屏大屏布局比手机多断点适配更简单、更稳定。

### 3.2 界面开发可行性

评估结论：高。

原因：

- 本阶段主要是卡片、列表、状态条、导航栏、表单、详情面板等常规 UI 能力。
- 智慧农业场景不依赖复杂 3D、地图或高难图形渲染。
- 无需真实数据接入时，可直接通过本地 mock 状态驱动页面。

### 3.3 交互演示可行性

评估结论：高。

原因：

- 提交中、成功、失败、离线等状态都可通过本地定时器与 mock service 模拟。
- 告警、指标变化、设备在线状态都能通过预设场景切换完成。
- 比起等真实网关，原型方式更适合早期做答辩和评审演示。

### 3.4 后续演进可行性

评估结论：中高。

条件：

- 前提是本阶段定义好 `models/`、`mock/`、`state/` 的边界。
- 如果页面直接写死业务数据，后续替换为真实服务会增加返工。

建议：

- 以“可替换 mock service”为前提设计原型代码结构。

## 4. 业务可行性评估

### 4.1 是否符合当前项目阶段

评估结论：非常符合。

原因：

- 当前没有业务代码，直接冲真实联调价值不高。
- 先做界面 demo 更容易统一认知、收敛范围、验证视觉方案。
- 该赛题又强调演示稳定性，因此优先把“看起来能跑通的主链路”建立起来很合理。

### 4.2 是否有助于后续比赛交付

评估结论：有明显帮助。

帮助体现在：

- 可提前验证首页和主路径是否适合演示。
- 可为 PPT、答辩稿、演示视频提供早期素材。
- 可帮助团队尽早暴露信息架构和交互设计问题。
- 可提前确定后续真实开发的页面骨架和数据结构。

## 5. 资源与周期可行性评估

### 5.1 人力要求

建议最小配置：

- 1 名主开发：负责 ArkUI 页面、组件与 mock 状态。
- 1 名协作者可选：负责视觉资源、文案优化、测试和演示流程校对。

评估：

- 即使单人推进，也具备较高可执行性。

### 5.2 周期建议

如果聚焦“先跑出高质量 demo”，建议节奏如下：

- 第 1-2 天：搭建壳体、导航、样式变量、mock 数据结构。
- 第 3-4 天：首页、灌溉页、补光页。
- 第 5 天：设备页、告警页、设置页。
- 第 6 天：场景切换、状态完善、视觉统一。
- 第 7 天：预览验证、问题修整、演示路径打磨。

评估：

- 一周左右可以完成一个较完整、可演示的原型版本。

## 6. 风险评估

### 6.1 高风险项

#### 风险 1：原型目标失焦

表现：

- 一边做原型，一边试图实现真实接口与真实业务逻辑，导致周期失控。

建议：

- 明确原型阶段只做“界面 + mock 演示 + 可演进结构”。

#### 风险 2：横屏布局缺少统一规范

表现：

- 每页独立开发后风格不统一，控件尺寸和布局节奏混乱。

建议：

- 先做全局控制台布局和设计变量，再做单页开发。

#### 风险 3：数据结构写死在页面中

表现：

- 后续联调时需要重写页面逻辑。

建议：

- 页面只消费统一模型和 mock service 输出。

### 6.2 中风险项

#### 风险 4：视觉过于普通，演示效果一般

建议：

- 在农业感、控制台感、告警层次感上做有辨识度的设计。

#### 风险 5：触控尺寸不足

建议：

- 在 `10.1` 寸场景下优先保证按钮和切换控件可点击，不要过度追求信息堆叠。

### 6.3 低风险项

#### 风险 6：页面数量较多

建议：

- 通过统一布局组件与卡片组件复用降低成本。

## 7. 可行性量化判断

| 维度 | 评级 | 说明 |
|---|---|---|
| 技术实现可行性 | 高 | ArkUI 常规页面与状态模拟，难度可控 |
| 平台适配可行性 | 高 | 已有 OH 工程，tablet 横屏方向明确 |
| 周期可控性 | 高 | 原型阶段不依赖真实联调 |
| 后续演进性 | 中高 | 前提是结构设计合理 |
| 展示价值 | 高 | 可快速形成可见成果 |
| 综合结论 | 高 | 建议立即执行 |

## 8. 启动建议

建议以“原型先行”的方式推进，并遵循以下启动顺序：

1. 统一横屏布局、主题变量和组件规范。
2. 首页先行，尽快形成能体现价值的首屏。
3. 优先完成灌溉和补光两条控制链路页面。
4. 补齐设备、告警、设置与演示场景切换。
5. 通过预览和真机验证优化触控体验。

## 9. 最终建议

综合判断：

- **建议立即启动原型 Demo 开发。**
- **建议把“设备/网关占位 + mock 场景切换 + 10.1 寸横屏控制台布局”作为本阶段主策略。**
- **建议本阶段只追求可演示、可扩展、可替换，不追求真实业务闭环。**
