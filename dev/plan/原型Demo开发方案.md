|---|---|

## 1. 方案目标

在当前仅有 OpenHarmony 初始化模板工程的基础上，先开发一个“界面先行”的智慧农业控制系统原型 Demo，用于完成以下目标：

- 验证核心页面信息架构是否合理。
- 验证 `10.1` 寸横屏触摸设备上的页面布局、信息密度和触控可达性。
- 形成可演示的首页、设备、灌溉、补光、告警、设置等主流程界面。
- 在不接入真实网关和设备的前提下，预演监测、控制、反馈、离线、告警等典型状态。
- 为后续接入真实服务、数据模型和联调逻辑预留清晰的替换位。

## 2. 原型阶段边界

### 2.1 本阶段要完成的内容

- 完成适配 `10.1` 寸横屏的整体 UI 框架。
- 完成首页总览、设备管理、灌溉控制、补光控制、告警中心、系统设置 6 个页面。
- 完成页面之间的导航切换和基本状态管理。
- 完成 mock 数据驱动下的动态展示。
- 完成典型状态演示：在线、离线、缺失、处理中、成功、失败、告警触发。
- 完成可用于答辩/评审展示的软件界面 demo。

### 2.2 本阶段不完成的内容

- 不接真实硬件、传感器、网关或云端服务。
- 不实现真实 HTTP/WebSocket 通信。
- 不实现完整数据库持久化。
- 不实现真实自动化规则引擎。
- 不以生产质量为目标处理全部异常和边角功能。

## 3. 原型 Demo 产品定位

本 Demo 是一个“可交互、可演示、可逐步演进”的 OpenHarmony 原生界面原型，不是静态设计稿，也不是完整业务系统。

它应满足以下要求：

- 评委或老师打开应用后，可以快速理解系统在做什么。
- 用户可以通过点击完成核心演示路径。
- 每个操作都能看到状态反馈，哪怕底层只是 mock。
- 页面结构后续可以平滑接入真实服务层，而不需要重做 UI 架构。

## 4. 目标设备与适配策略

### 4.1 目标设备假设

- 设备尺寸：`10.1` 寸。
- 主要使用方式：固定安装触摸屏。
- 主要方向：横屏。
- 触控距离：中近距离查看与操作。
- 主要场景：展台演示、小型温室控制台、教学展示。

### 4.2 适配原则

- 默认横屏优先设计，竖屏仅做最低限度兼容，不作为主验收方向。
- 采用“大区块 + 中高信息密度 + 明确触控区”的控制台式布局。
- 关键按钮满足触控点击面积，避免过小交互元素。
- 核心信息尽量首屏可见，减少页面深层跳转。
- 保证 1280x800 一类分辨率下视觉均衡，同时兼容近似比例平板。

### 4.3 建议布局规则

- 顶部保留全局状态栏：系统名称、网关状态、当前时间、告警摘要。
- 左侧保留一级导航栏，支持大按钮触控切换。
- 右侧作为主内容区，按页面展示卡片、列表和控制面板。
- 首页采用 `2-3` 列卡片式仪表盘布局。
- 控制页优先采用左右分栏：左侧状态和说明，右侧控制面板。

## 5. 原型 Demo 页面范围

### 5.1 首页总览页

目标：3 秒内让用户理解当前环境与设备状态。

建议内容：

- 网关状态卡片。
- 温度、湿度、光照、二氧化碳、土壤湿度 5 个核心指标卡。
- 在线设备数/离线设备数统计。
- 最近告警摘要。
- 快捷操作入口：灌溉、补光、设备、告警、设置。

演示状态：

- 正常数据。
- 部分数据缺失。
- 网关离线。
- 告警高亮。

### 5.2 设备管理页

目标：展示设备清单和设备状态，为后续接入真实设备模型打基础。

建议内容：

- 设备列表卡片。
- 设备类型标签：PUMP、LED、FAN、SENSOR。
- 在线/离线状态。
- 当前状态与位置。
- 设备详情侧边区域。

演示状态：

- 设备在线可控。
- 设备离线禁用控制。
- 扩展设备占位展示。

### 5.3 灌溉控制页

目标：展示土壤湿度与灌溉控制流程。

建议内容：

- 当前土壤湿度卡片。
- 灌溉建议说明区。
- 灌溉时长设置控件。
- 开始/停止按钮。
- 当前控制状态条。
- 最近操作记录区。

演示状态：

- submitting。
- accepted/success。
- failed。
- 设备离线。

### 5.4 补光控制页

目标：展示光照强度与补光调节界面。

建议内容：

- 当前光照值卡片。
- LED 开关。
- 亮度等级调节。
- 自动补光开关占位。
- 规则说明占位。
- 最近补光记录。

演示状态：

- 手动补光成功。
- 亮度切换中。
- 自动模式占位开启。

### 5.5 告警中心页

目标：展示异常监测能力和可解释性。

建议内容：

- 告警列表。
- 告警等级筛选。
- 告警状态筛选。
- 告警详情面板。
- 确认处理按钮。

演示状态：

- NEW。
- ACK。
- RESOLVED。

### 5.6 系统设置页

目标：为未来网关接入、规则配置、扩展设备管理预留入口。

建议内容：

- 网关地址与连接状态占位。
- 数据刷新模式占位。
- 告警阈值配置占位。
- 自动规则配置占位。
- 演示模式开关。

说明：

- 本页以“配置表单演示”和“后续扩展位展示”为主，不追求完整实现。

## 6. 技术实现规划

### 6.1 技术路线

- 技术栈：`Stage 模型 + UIAbility + ArkTS + ArkUI`。
- 保持单 `UIAbility` + 多页面/多视图结构。
- 本阶段不引入重量级第三方框架。
- 优先使用组件化 + 本地 mock 状态驱动方式实现。

### 6.2 推荐目录规划

建议在 `entry/src/main/ets/` 下逐步形成如下结构：

```text
entry/src/main/ets/
├─ app/
│  ├─ App.ets
│  ├─ routes.ets
│  └─ constants/
├─ pages/
│  ├─ DashboardPage.ets
│  ├─ DevicePage.ets
│  ├─ IrrigationPage.ets
│  ├─ LightingPage.ets
│  ├─ AlertPage.ets
│  └─ SettingsPage.ets
├─ components/
│  ├─ layout/
│  ├─ cards/
│  ├─ charts/
│  ├─ device/
│  └─ feedback/
├─ mock/
│  ├─ mockData.ets
│  ├─ mockGatewayService.ets
│  └─ mockScenarioController.ets
├─ models/
│  ├─ device.ets
│  ├─ telemetry.ets
│  ├─ alert.ets
│  └─ command.ets
└─ state/
   ├─ appState.ets
   └─ demoState.ets
```

### 6.3 分层建议

- `pages/`：页面编排和交互入口。
- `components/`：可复用卡片、列表项、状态条、导航组件。
- `mock/`：全部占位数据和伪服务逻辑。
- `models/`：统一类型结构，减少后续替换成本。
- `state/`：集中管理当前演示状态、页面切换和场景切换。

## 7. Demo 交互策略

### 7.1 交互目标

- 让观众明确感知“系统正在监测和控制设备”。
- 让关键操作都具有即时反馈。
- 在没有真实设备时，也能演示“提交中 -> 成功/失败”的过程。

### 7.2 场景切换建议

建议内置“演示模式场景面板”，可快速切换以下预设：

- 场景 A：正常运行。
- 场景 B：土壤湿度过低，建议灌溉。
- 场景 C：光照过低，建议补光。
- 场景 D：网关离线。
- 场景 E：设备离线。
- 场景 F：高温/高 CO2 告警。

这样可以在评审时快速演示，不依赖临时造数据。

## 8. 视觉与交互建议

### 8.1 视觉方向

- 整体采用“农业控制台”风格，而不是通用后台管理风格。
- 主色建议使用绿色、土壤棕、光照暖黄、告警橙红的组合。
- 背景不宜过花，应以浅纹理、渐变和大卡片层次为主。
- 指标卡片需具备良好远距可读性。

### 8.2 控件建议

- 一级导航按钮采用大尺寸图标 + 文本。
- 关键控制按钮要有明显主次区分。
- 滑杆、开关、标签等控件优先选择触控友好样式。
- 告警颜色需保持醒目但不刺眼。

## 9. 开发里程碑建议

### 里程碑 1：基础壳与导航

- 完成横屏整体布局。
- 完成左导航 + 顶部状态栏。
- 完成多页面切换框架。

产出：可运行的应用骨架。

### 里程碑 2：首页与核心卡片

- 完成首页环境指标卡片。
- 完成设备摘要和告警摘要。
- 完成正常/异常/离线状态切换。

产出：可展示系统价值的首页原型。

### 里程碑 3：控制页面

- 完成灌溉页。
- 完成补光页。
- 实现控制按钮、状态反馈和记录占位。

产出：可演示控制闭环的交互页面。

### 里程碑 4：告警与设置页面

- 完成告警列表/详情。
- 完成设置页占位表单。
- 完成演示场景切换控制。

产出：完整 demo 页面集合。

### 里程碑 5：视觉统一与演示优化

- 统一样式变量。
- 优化卡片间距、字体层级、触控反馈。
- 修整演示路径和状态切换。

产出：可录屏、可答辩、可现场演示版本。

## 10. 验收标准

### 10.1 原型阶段验收

- 应用可在 OpenHarmony 工程中成功运行。
- 横屏下首页、设备、灌溉、补光、告警、设置页面可访问。
- 页面布局适配 `10.1` 寸横屏，首屏不拥挤、不失衡。
- 关键交互可点击，且有明确反馈。
- 无真实设备时可通过 mock 场景完成主演示路径。

### 10.2 演示路径验收

- 可从首页进入灌溉页并完成一次“开始灌溉”演示。
- 可从首页进入补光页并完成一次亮度调节演示。
- 可从首页或告警页展示一次异常告警。
- 可切换到网关离线或设备离线场景。

## 11. 风险与预留策略

### 11.1 当前主要风险

- 如果直接追求完整业务实现，会显著拖慢界面 demo 交付。
- 若页面结构和 mock 数据模型不统一，后续接真服务时会返工。
- 若横屏布局没有先统一框架，后续各页面会风格割裂。

### 11.2 风险控制策略

- 先统一布局骨架和设计变量，再进入单页开发。
- 先定义 mock 模型结构，尽量贴近后续真实接口字段。
- 把“演示场景切换”做成明确机制，避免把状态写死在页面里。

## 12. 下一步建议

按优先级建议立即开展：

1. 先搭建横屏控制台壳体与路由骨架。
2. 再完成首页和两大控制页。
3. 然后补齐告警页、设置页和演示场景切换。
4. 最后统一视觉风格并进行真机/预览适配验证。
