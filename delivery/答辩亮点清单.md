# 答辩亮点清单

## 1. 项目定位亮点

- 项目不是单纯页面展示，而是“监测 + 控制 + 告警 + 联动 + 扩展”的完整系统。
- 方案从赛题文本出发，补齐了需求、设计、测试和交付链路。

## 2. 技术路线亮点

- 采用 OpenHarmony 原生路线：`Stage 模型 + UIAbility + ArkTS + ArkUI`。
- 避免单纯 Web 套壳，更贴合题意和平台能力。
- 采用 `Preferences + RDB` 组合兼顾配置存储和历史记录。

## 3. 系统架构亮点

- 采用“设备层 - 网关层 - 应用层 - 展示层”四层架构。
- 通过网关统一协议适配，降低 APP 与硬件协议耦合。
- 支持后续新增设备和新增规则，不需要推翻整体架构。

## 4. 功能亮点

- 首页总览可快速展示关键环境状态。
- 可完成灌溉控制和补光控制闭环。
- 可配置阈值告警和自动联动规则。
- 支持扩展设备接入，证明平台具备扩展能力。

## 5. 工程化亮点

- 文档体系完整，不只停留在功能描述层。
- 具备接口定义、数据库设计、测试方案、演示脚本等工程化材料。
- 需求、设计、测试之间具备追踪关系。

## 6. 可演示性亮点

- 设计了完整演示脚本和 PPT 提纲。
- 关键演示环节可稳定复现：看数、控设备、出告警、跑联动、接扩展设备。
- 提供了 mock 与真实设备双路径兜底思路。

## 7. 可扩展性亮点

- 设备模型统一，扩展风扇只是开始，后续还可接遮阳帘、加热器、摄像头等。
- 联动规则模型支持“条件-动作”扩展。
- 历史记录、区域化管理、图表增强都可以作为下一阶段演进方向。

## 8. 回答高频问题建议

### 8.1 为什么不用 React 做主界面？

建议回答：

- 我们评估过 React 可行性，但赛题更强调 OpenHarmony 原生适配和系统能力结合，因此主路线选择 ArkTS + ArkUI；如果后续做复杂图表页，可以再局部引入 Web 能力。

### 8.2 为什么要用网关抽象？

建议回答：

- 因为底层设备协议可能多样，如果 APP 直接对接每种设备会让系统耦合度很高。通过网关抽象后，APP 面向统一接口开发，更利于比赛周期内快速落地和后续扩展。

### 8.3 你们的系统亮点是什么？

建议回答：

- 我们不仅做了界面，还完成了从需求、架构、接口、数据库、测试到演示的全链路方案设计，重点体现了控制闭环、自动规则和扩展能力。
