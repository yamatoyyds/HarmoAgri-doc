## 1. 文档目的

本文档用于识别当前版本在硬件、接口、范围、测试和交付上的主要风险，并给出升级条件、责任人和范围收缩策略。

## 2. 主要风险

### 风险 1：硬件不可用或联调延迟

- 影响：控制闭环无法及时验证，扩展设备接入受阻
- 应对：优先以 mock 网关和模拟设备打通首页与控制主路径

### 风险 2：接口变更频繁

- 影响：前端和网关联调反复返工，测试口径失效
- 应对：先冻结统一字段模型、错误结构和接口契约

### 风险 3：功能过多导致周期失控

- 影响：核心功能不稳定，演示主路径失败
- 应对：坚持 P0 + MVP 优先，P1 视进度和硬件情况后置

### 风险 4：测试和文档滞后

- 影响：交付质量下降，需求和实现不一致
- 应对：每阶段同步维护 PRD、追踪矩阵、测试方案和测试案例

### 风险 5：主演示路径不稳定

- 影响：即使功能数量足够，也无法形成高质量交付
- 应对：优先保障“首页看数 -> 手动灌溉 -> 手动补光/告警”路径稳定

## 3. 风险优先级

| 风险 | 概率 | 影响 | 优先级 |
|---|---|---|---|
| 硬件联调延迟 | 高 | 高 | 高 |
| 接口频繁变化 | 中 | 高 | 高 |
| 周期失控 | 中 | 高 | 高 |
| 文档滞后 | 高 | 中 | 中 |
| 主演示路径不稳定 | 中 | 高 | 高 |

## 4. 风险应对机制

### 4.1 每周风险复盘

- 每周至少一次检查硬件、接口、功能、测试和文档风险
- 对高优先级风险指定责任人和缓解动作
- 复盘结果必须同步影响开发计划和范围基线

### 4.2 风险升级条件

- P0 功能连续两天无推进
- 首页、灌溉、补光三条主路径任一无法闭环
- 核心硬件持续不可用
- 网关协议连续变更导致应用返工
- TC-19 主演示链路无法稳定复现

发生上述情况时，应立即启动范围收缩策略。

## 5. 范围收缩策略

### 5.1 保底范围

- 首页总览
- 网关状态显示
- 手动灌溉控制
- 手动补光控制

### 5.2 MVP 范围

- 首页总览
- 网关状态显示
- 设备列表
- 手动灌溉控制
- 手动补光控制
- 基础告警
- 1 类扩展设备

### 5.3 可后置范围

- 自动补光
- 联动规则
- 历史记录展示
- 多设备区域化展示
- 视觉动画和高级美化

## 6. 降级交付策略

- 硬件风险：以 mock 网关替代真实网关，优先保住主路径演示
- 扩展设备风险：以模拟风扇保留扩展能力说明
- 推送风险：以 HTTP 轮询作为 WebSocket 的兜底
- P1 进度风险：将自动补光和联动规则改为说明性展示或后置项

## 7. 风险责任建议

| 风险类型 | 建议责任角色 |
|---|---|
| 硬件联调风险 | 网关/设备联调负责人 |
| 接口变更风险 | 网关负责人 + 应用负责人 |
| 周期风险 | 项目负责人 |
| 文档风险 | 产品/文档负责人 |
| 测试风险 | 测试负责人 |
| 主路径演示风险 | 项目负责人 + 测试负责人 |

## 8. 变更与同步要求

- 一旦范围收缩，必须同步修改 PRD、开发计划、测试方案和需求追踪矩阵
- 一旦接口契约变化，必须同步修改接口设计、字段字典和测试案例
