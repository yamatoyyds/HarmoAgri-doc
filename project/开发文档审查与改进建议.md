## 1. 文档目的

本文档用于从严格产品经理视角审查当前 `doc/` 目录下的开发文档质量，明确：

- 当前文档体系已经具备哪些可用基础

- 哪些改进项应优先纳入后续文档修订计划

本文档定位不是替代现有 PRD、设计文档和测试文档，而是作为后续改进参考和评审基线。

## 3. 总体结论

### 3.1 正向评价

当前文档体系已经具备较好的基础框架，主要优点如下：

- 文档分层较完整，已经覆盖产品、设计、数据库、测试、交付和项目管理。
- 项目主线清晰，围绕“监测 + 控制 + 告警 + 联动 + 扩展设备”展开。
- 技术路线相对统一，围绕 OpenHarmony 原生方案展开，方向明确。
- 需求、设计、接口、数据库和测试之间已经建立了初步映射关系。
- 对比赛导向和交付导向有较强意识，已考虑演示、答辩和范围收缩。

### 3.2 总体判断

当前文档状态可以支持项目启动开发，但还没有完全达到“高确定性产品交付文档”的标准。

更准确地说，当前文档体系已经完成了“骨架搭建”，但尚未完成“交付级内容填充”。

目前多数文档已经回答了“应该写哪些章节”，但还没有充分回答“每个章节需要写到什么细度，才能真正指导开发、联调、测试、验收和答辩”。

主要问题不在于“文档数量不足”，而在于以下几个方面仍然不够扎实：

- 产品目标不够量化
- 版本边界不够严格
- 用户价值表达不够聚焦
- 验收标准不够可测试
- 需求到测试的闭环还不够严密

如果不进一步修订，后续很容易出现以下情况：

- 团队对优先级理解不一致
- 开发过程中边做边改需求
- 联调阶段接口和字段解释不统一
- 测试通过与否缺少客观标准
- 最终可演示，但难以证明“产品方案成熟且完成度高”

因此，本次审查的核心结论是：

- 当前文档是“骨架级文档”，不是“交付级文档”
- 后续修订重点不应放在继续补章节，而应放在扩充细节、落实细节、固化细节
- 判断文档是否合格的标准，不应再停留在“结构完整”，而应提升到“是否足以直接指导实施”

### 3.3 后续修订的核心原则

后续所有文档修订，都建议围绕以下原则展开：

#### 原则 1：从“有章节”转向“有内容深度”

- 不能只满足于文档目录完整
- 必须让每个章节都有足够支撑实施的细节
- 重点不是“写到了”，而是“写透了”

#### 原则 2：从“描述方向”转向“约束落地”

- 不仅要写目标，还要写成功标准
- 不仅要写功能，还要写边界、异常、限制和失败处理
- 不仅要写流程，还要写前置条件、后置结果和判定口径

#### 原则 3：从“看起来完整”转向“可以直接使用”

每一类文档都应经得起以下问题检验：

- 这段内容能否直接指导开发实现？
- 这段内容能否直接支撑接口联调？
- 这段内容能否直接转成测试用例？
- 这段内容能否直接用于验收判定？

如果以上问题不能明确回答“能”，说明文档细节仍然不够。

#### 原则 4：修订重点放在扩充和落实细节，而不是继续堆骨架

后续修订时，应明确避免以下低效动作：

- 继续新增大量标题但缺少实质内容
- 用抽象表述替代执行细节
- 用原则性语言代替可验证标准

应优先推进以下动作：

- 扩充字段、状态、交互、错误处理等执行细节
- 落实优先级、范围边界、验收标准、异常规则
- 固化开发、联调、测试三方共用的统一口径

## 4. 关键问题清单

## 4.1 P0 级问题：必须优先修复

### 问题 1：产品目标不够业务化和量化

当前 PRD 对目标的描述偏功能型和交付型，缺少明确的版本成功标准。

典型表现：

- 强调“要实现哪些功能”，但缺少“本版本最重要的成功结果是什么”
- 缺少可量化的成功指标，例如主演示链路成功率、首屏展示时长、控制反馈时延
- 缺少“为什么先做这些、不做哪些”的强约束

影响：

- 团队容易把所有需求都当成重要需求
- 难以在资源有限时做清晰取舍
- 答辩时不容易讲清产品价值和版本目标

改进建议：

- 在 PRD 中新增“版本目标与成功标准”章节
- 明确 V1 的唯一主目标，例如：稳定完成环境监测、手动控制、状态反馈闭环，并验证设备扩展性
- 为 P0 能力增加可量化指标，例如：首屏 3 秒内可见、控制 3 秒内反馈、演示主链路一次通过

### 问题 2：目标用户定义混杂，真实用户与评审对象未分离

当前文档同时把农户、管理员、评委都放在目标用户体系内，这会混淆真实用户需求与展示诉求。

影响：

- 容易为了“给评委看”而引入并不属于真实使用场景的需求
- 功能优先级判断容易被演示偏好干扰
- 用户故事会变得不够真实

改进建议：

- 将“农户/操作员”和“管理员”保留为真实用户角色
- 将“评委/观察者”单独归入“关键利益相关方”
- 单独增加“答辩关注点”或“演示关注点”章节，而不是并入用户角色

### 问题 3：MVP、保底范围、增强范围尚未形成统一版本基线

当前文档中存在 MVP、保底范围、增强范围等表述，但这些表述分散在不同文档中，尚未形成统一口径。

影响：

- 版本边界容易漂移
- 团队对哪些能力必须做、哪些能力可后置理解不同
- 风险发生时难以及时收缩范围

改进建议：

- 在 PRD 或项目管理文档中新增统一的“版本范围基线表”
- 按 `MVP / 保底 / 增强 / 不做` 四层拆分所有需求
- 保证 `PRD`、`项目详细方案`、`风险管理`、`开发计划` 四份文档的版本范围口径一致

### 问题 4：验收标准偏主观，不够可测试

目前不少验收标准使用了“可正常展示”“可正常执行”“具备能力”这样的表达，测试时解释空间过大。

影响：

- 测试结论容易主观化
- 不同人员对“通过”的理解不一致
- 难以在答辩或评审时证明完成质量

改进建议：

- 所有 P0 需求改写为“可观察、可测量、可验证”的验收条件
- 示例：
  - 不建议写：首页可正常展示核心环境数据
  - 建议写：首页首屏 3 秒内展示至少 5 类环境指标，缺失项显示 `--`，且不阻塞其他卡片渲染
- 所有控制型需求都补充时延、失败提示、状态回传条件

## 4.2 P1 级问题：重要问题，应在近期修复

### 问题 5：竞品分析偏抽象，未真正支撑产品取舍

当前竞品分析已经有结构，但更像背景说明，尚未真正转化为版本决策依据。

主要不足：

- 竞品对象偏泛化，没有更贴近真实可参考方案
- 对比维度有，但未得出“因此本项目应该优先做什么、放弃什么”
- 差异化设计写得较概括，缺少与需求优先级的联动

改进建议：

- 将竞品分析输出直接映射到需求取舍
- 在每个竞品分析后补一段“对本项目的启示”
- 明确说明为什么本项目优先做闭环、扩展设备、原生鸿蒙适配，而不是复杂后台报表或重型云平台能力

### 问题 6：关键体验路径存在，但“第一优先演示路径”不够突出

当前文档中已经有 KEP，但没有进一步明确：

- 哪条路径是答辩时必须一次成功的主演示路径
- 哪条路径是增强项
- 资源受限时哪些场景必须保留

改进建议：

- 对 KEP 重新分层：主演示路径、辅助演示路径、增强演示路径
- 至少固化 3 条一镜到底路径：
  - 首页看数
  - 手动灌溉控制
  - 告警或自动补光触发
- 每条路径增加前置条件、失败兜底、验收标准

### 问题 7：历史记录需求状态不够稳定

部分文档中将历史记录视为需求，部分文档又将其视为增强项，产品定义不够统一。

影响：

- 容易导致开发时不清楚是否要投入实现
- 测试侧无法判断是否必须覆盖
- 页面规格和数据库设计容易出现冗余或不足

改进建议：

- 在 PRD 中明确历史记录在 V1 的定位：
  - 若属于 P2，则说明仅保留轻量展示，不要求复杂筛选和趋势分析
  - 若属于 P1，则明确展示范围、保留时长和核心价值

### 问题 8：规则能力定义偏工程化，产品规则不够明确

当前规则服务更多描述了技术结构，但缺少产品层的规则约束。

缺失点包括：

- 手动与自动规则冲突时谁优先
- 连续触发如何去抖
- 同一设备是否允许并发动作
- 规则失败后是否重试
- 规则是否需要冷却时间

改进建议：

- 单独补一节“规则产品规则说明”
- 对每种冲突场景给出唯一处理原则
- 让测试用例与这些规则一一对应

### 问题 9：接口契约还不够完整

当前接口文档已经具备草案基础，但距离联调级契约仍有差距。

主要缺失：

- 统一错误响应结构
- 字段是否允许缺失或为空的明确约束
- 分页结果统一结构
- 接口超时和重试策略
- 鉴权现状与扩展方案边界

改进建议：

- 为每个 REST 接口补充成功响应和失败响应示例
- 增加统一错误响应格式，例如 `code`、`message`、`requestId`、`details`
- 标注字段是否 nullable、默认值是什么、缺失时前端如何处理

### 问题 10：需求追踪矩阵还不够完整

当前追踪矩阵已经建立，但覆盖尚不完整。

主要问题：

- 并非所有 FR 都完整映射到设计、接口、数据表和测试
- NFR 基本没有进入正式追踪矩阵
- 交付材料的映射仍偏粗

改进建议：

- 每个 FR/NFR 都建立唯一映射行
- 增加字段：优先级、版本归属、是否阻塞演示
- 对非功能需求补齐测试映射，例如性能冒烟、异常恢复、稳定性验证

## 4.3 P2 级问题：建议优化项

### 问题 11：页面规格更像信息架构，交互决策仍不够细

当前页面规格已经说明区块和字段，但关键交互规则仍不够细化。

建议补充：

- 控制动作是否二次确认
- 失败提示文案策略
- 禁用态和离线态的视觉区分
- 自动模式与手动模式共存时的页面提示方式
- 空态、错误态、离线态、部分成功态的统一表现规则

### 问题 12：数据字典覆盖面偏小

当前数据字典只覆盖了部分关键枚举，仍缺少很多业务字段和状态定义。

建议补充：

- `metricType` 全量枚举
- `resultStatus` 全量枚举
- 设备状态与设备能力枚举
- 规则启停状态
- 页面状态字典
- 错误码来源与适用接口范围

### 问题 13：项目计划缺少更强的资源约束视角

当前计划已经列出阶段和周安排，但对依赖和资源占用描述还不够细。

建议补充：

- 每阶段责任人
- 进入条件与退出条件
- 若硬件未就绪时的替代路线
- 每周必须交付的可验收输出物

### 问题 14：缺少正式的需求变更机制

当前虽然有风险管理，但没有形成正式的需求变更机制。

建议补充：

- 谁可以提出需求变更
- 谁负责批准
- 是否影响当前版本基线
- 是否需要同步修改测试用例和追踪矩阵

## 5. 推荐修订优先顺序

建议按以下顺序推进修订：

修订时需要始终坚持一个前提：不是把文档“补得更多”，而是把文档“写得更能落地”。

同时需要特别强调：

- 本次审查意见不是只针对 `PRD`
- `PRD` 只是修订起点，不是修订终点
- 当前问题是整套文档普遍停留在骨架层，因此必须联动修订，而不能只修一份文档

如果只修改 `PRD`，而接口、页面、测试、追踪矩阵等文档不跟着细化，最终仍然会出现以下问题：

- 产品目标写清了，但设计实现细节仍然模糊
- 需求口径更新了，但接口和字段定义没有同步
- 页面行为补充了，但测试用例仍然无法准确验证
- 需求有了验收标准，但追踪关系没有形成闭环

因此，建议把本轮修订视为一次“整套文档从骨架到可交付”的联动整改，而不是单点优化。

## 8. 最终结论

当前 `doc/` 文档体系已经具备较好的“开发启动条件”，但距离“严格产品管理下的高质量交付文档”仍有差距。

更明确地说，当前文档的主要问题不是没有骨架，而是只有骨架；不是没有方向，而是细节尚未落实到足以支撑交付的程度。

后续修订的重点不应是继续堆叠新文档，而应优先做以下四件事：

- 压实产品目标
- 冻结版本边界
- 量化验收标准
- 打通需求到测试的完整追踪关系

同时必须明确一个修订导向：后续工作重心应从“补框架”切换到“补细节、补约束、补标准、补边界”。

只有当文档中的页面、接口、规则、数据、测试、风险和验收标准都细化到可直接指导实施时，这套文档才能从“骨架级”真正升级为“交付级”。

如果这四件事完成，现有文档体系将会从“方案可读”明显提升为“开发可执行、联调可收口、测试可判定、答辩可说明”。
