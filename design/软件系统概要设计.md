## 1. 设计目标

软件系统需要满足以下目标：

- 页面结构清晰且与 MVP 范围一致
- 业务逻辑模块化，支持主路径优先实现
- 设备控制与状态同步解耦
- 页面状态、接口状态和持久化记录保持一致
- 便于测试和后续迭代

## 4. 软件分层

```text
UI 层
  ↓
State / ViewModel 层
  ↓
Domain Service 层
  ↓
Repository 层
  ↓
Network / Local DB 层
```

分层约束：

- UI 层不直接调用数据库
- 页面不直接感知网关协议细节
- Repository 层只负责数据来源统一，不承担业务决策
- Domain Service 层负责规则、控制和异常处理逻辑

## 5. 核心流程摘要

### 5.1 首页展示流程

1. 页面进入时请求网关状态和实时数据
2. GatewayRepository 返回最新环境信息
3. DeviceRepository 返回设备摘要
4. Repository 写入缓存或数据库
5. 页面根据状态渲染 `ready`、`partial_error` 或 `offline`

### 5.2 控制执行流程

1. 用户触发控制动作
2. Domain Service 校验设备状态和参数
3. GatewayService 下发命令
4. 页面进入 `submitting`
5. 接收 `accepted` 和最终状态推送
6. 更新页面状态并写入操作日志

### 5.3 规则执行流程

1. 实时数据到达
2. RuleService 判断规则条件是否满足
3. 如满足则生成动作指令
4. 调用目标服务执行控制逻辑
5. 写入规则执行结果和控制日志

## 6. 模块间依赖约束

| 模块 | 允许依赖 | 不允许依赖 |
|---|---|---|
| 页面模块 | State 层、Service 层 | Repository、DB 直接访问 |
| Service 层 | Repository、领域模型 | UI 组件 |
| Repository 层 | Network、Preferences、RDB | UI 和页面状态 |

## 7. 页面状态与服务状态对应

| 页面状态 | 服务侧含义 |
|---|---|
| loading | 正在请求首屏数据 |
| ready | 数据完整可展示 |
| partial_error | 部分数据缺失但可降级 |
| offline | 网关不可达 |
| submitting | 控制或保存请求已发送 |
| success | 最终操作成功 |
| failed | 最终操作失败 |
