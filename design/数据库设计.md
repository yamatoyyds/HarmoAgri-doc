# 数据库设计

## 1. 设计目标

数据库用于保存以下数据：

- 设备信息
- 传感器历史数据
- 告警规则和告警记录
- 联动规则
- 控制日志
- 系统配置缓存

## 2. 存储策略

### 2.1 Preferences

适合保存：

- 网关地址
- 用户偏好设置
- 页面展示偏好
- 最近连接信息

### 2.2 RDB

适合保存：

- 设备表
- 传感器记录表
- 告警规则表
- 告警事件表
- 联动规则表
- 控制命令记录表

## 3. 数据表关系

```text
device
├─ 1..n sensor_reading
├─ 1..n command_record
└─ 1..n alert_event

alert_rule
└─ 1..n alert_event

automation_rule
└─ 1..n command_record
```

## 4. 主键与时间设计

- 主键统一采用字符串 ID 或自增整型。
- 时间字段统一采用毫秒时间戳或 ISO 时间字符串。
- 删除策略优先逻辑保留，避免误删历史记录。

## 5. 分表与索引建议

### 5.1 索引建议

| 表名 | 索引字段 | 原因 |
|---|---|---|
| `device` | `type`, `online_status` | 支持设备筛选 |
| `sensor_reading` | `device_id`, `metric_type`, `collected_at` | 支持按时间查询趋势 |
| `alert_event` | `level`, `status`, `created_at` | 支持告警页筛选 |
| `command_record` | `device_id`, `created_at` | 支持控制历史查询 |

### 5.2 数据保留建议

- `sensor_reading`：保留近 30 天原始记录，超期可归档或按日聚合。
- `command_record`：保留全部比赛周期记录。
- `alert_event`：保留全部比赛周期记录，便于答辩展示。

## 6. 数据一致性策略

### 6.1 设备状态一致性

- 设备在线状态以网关最新状态为准。
- 本地数据库只用于缓存和历史追踪，不作为最终真实来源。

### 6.2 控制记录一致性

- 命令发起后先写入 `ACCEPTED` 或 `PENDING` 状态。
- 收到网关最终结果后再更新为 `SUCCESS` 或 `FAILED`。

### 6.3 告警一致性

- 告警事件与规则定义分离。
- 规则修改后不回写历史事件，只影响未来触发。

## 7. 推荐查询场景

| 场景 | 主要表 |
|---|---|
| 首页展示最近数据 | `sensor_reading`, `device`, `alert_event` |
| 设备页展示当前状态 | `device`, `command_record` |
| 告警页展示历史与筛选 | `alert_event`, `alert_rule` |
| 规则页展示配置 | `alert_rule`, `automation_rule` |
| 演示报告取数 | `command_record`, `alert_event`, `sensor_reading` |
