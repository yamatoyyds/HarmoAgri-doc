## 1. 文档目的

本文档用于明确智慧农业控制系统 V1 的产品目标、版本边界、需求范围、关键体验路径和验收标准，作为设计、开发、联调、测试和答辩的统一输入。

本文档重点解决以下问题：

- V1 到底要完成什么
- 哪些内容必须做，哪些内容可以后置
- 什么叫“做到可交付”
- 需求如何映射到页面、接口、数据和测试

## 2. 需求背景

### 2.1 需求来源

本需求来源于 `doc/DOC_target.md` 中的企业赛题命题。题目要求基于 OpenHarmony 操作系统设计并开发一款智慧农业控制 APP，实现对农业环境数据的远程监测，以及对执行设备的远程控制。

### 2.2 行业背景

农业生产长期受劳动力老龄化、现场巡检频繁、环境感知滞后和资源投入不精准等问题影响。智慧农业通过传感器、物联网、数据分析与自动控制，可提升环境感知能力和控制效率，降低重复巡检成本。

### 2.3 项目背景

本项目同时具备以下三类属性：

- 比赛交付项目：需要在有限周期内形成完整文档和可演示系统
- 产品验证项目：需要证明 OpenHarmony 在智慧农业场景中的可行性
- 控制闭环项目：重点不是静态页面展示，而是监测、控制、反馈的完整链路

## 3. 产品概述

### 3.1 产品定位

本产品是一款面向小型温室、教学实验和比赛演示场景的 OpenHarmony 原生智慧农业控制应用，用于展示环境监测、设备控制、告警与基础自动化能力。

### 3.2 V1 聚焦场景

V1 不追求完整农业平台能力，重点聚焦以下单一场景：

- 在一个可演示的小型农业环境中，完成环境数据查看、手动控制、异常提示和基础扩展能力证明。

V1 不聚焦以下场景：

- 大规模多温室统一管理
- 长周期生产数据分析
- 重型后台管理与报表系统
- 多角色权限平台化运营

### 3.3 真实用户角色

#### 角色 A：农户/操作员

- 关注当前环境是否异常
- 需要快速执行灌溉和补光操作
- 需要明确知道控制是否成功

#### 角色 B：系统管理员

- 负责网关配置和设备接入
- 负责阈值规则和联动规则配置
- 负责定位系统离线、设备离线和规则异常问题

### 3.4 关键利益相关方

以下对象不是产品真实用户，但会影响交付标准：

- 指导教师：关注方案完整性和实现合理性
- 赛题评委：关注演示稳定性、闭环完整性、扩展能力和 OpenHarmony 贴合度
- 项目负责人：关注文档闭环、交付节奏和风险收缩能力

### 3.5 核心价值

- 用户打开首页后可在短时间内判断环境是否正常
- 用户可对关键设备执行手动控制并看到明确反馈
- 系统可对异常环境进行提示，并支持基础自动化能力展示
- 系统可接入至少 1 类扩展设备，证明后续扩展能力

## 4. 版本目标与成功标准

### 4.1 V1 目标

在 OpenHarmony 设备上稳定完成智慧农业系统的核心监测与控制闭环，满足比赛演示与文档交付要求。

### 4.2 成功标准

- 首页首屏进入后 3 秒内展示至少 5 项核心环境指标
- 首页可展示网关在线状态、在线设备数和告警摘要
- 手动灌溉和手动补光均可在 3 秒内得到明确反馈，反馈状态包括成功、失败或处理中
- 网关离线和设备离线场景均有明确提示，且应用不崩溃
- 至少 3 类阈值规则可配置并触发告警
- 至少 1 类扩展设备可展示并完成基础控制
- 主演示路径连续执行一次通过

## 5. 竞品方案分析

### 5.1 竞品识别

- 竞品 A：通用 IoT 智能温室控制平台
- 竞品 B：基于 Web 控制台的农业监测系统
- 竞品 C：轻量移动园艺控制应用

### 5.2 竞品能力分析

| 竞品 | 优势 | 不足 | 对本项目的启示 |
|---|---|---|---|
| 竞品 A | 架构成熟，自动化能力强，扩展性好 | 成本高，依赖云平台，复杂度高 | 本项目应保留分层和扩展思路，但必须控制实现复杂度 |
| 竞品 B | 图表展示强，后台能力成熟 | 原生系统能力弱，控制体验一般 | 本项目不应把重心放在复杂报表，而要优先保证原生控制闭环 |
| 竞品 C | 交互轻量，演示友好，控制路径直观 | 扩展能力弱，多设备建模不足 | 本项目应保留演示友好性，同时补足规则和扩展能力 |

### 5.3 差异化设计结论

基于竞品分析，V1 明确做以下取舍：

- 优先做原生 OpenHarmony 终端能力展示，不做复杂 Web 后台
- 优先做监测、控制、反馈闭环，不做大而全数据分析
- 优先做 1 类扩展设备接入，不做多类型复杂扩展管理
- 优先保证主演示路径稳定，不为增强项牺牲核心链路可靠性

## 6. 版本范围基线

### 6.1 范围分层

| 分层 | 需求项 | 说明 |
|---|---|---|
| MVP | 首页看数、网关状态、设备列表、手动灌溉、手动补光、基础告警、扩展设备接入 | 必须完成 |
| 保底 | 首页看数、网关状态提示、手动灌溉、手动补光 | 硬件风险发生时仍需保留 |
| 增强 | 自动补光、联动规则、历史记录 | 有余力时实现 |
| 不做 | 重型后台管理、复杂报表、多温室平台化管理、多角色权限系统 | V1 不纳入 |

### 6.2 范围冻结原则

- P0 需求未经项目负责人确认不得下调
- P1 需求在硬件或周期风险出现时可后置
- P2 需求不应阻塞 P0/P1 开发、联调和测试
- 新增需求必须同步更新需求追踪矩阵和测试用例

## 7. 用户故事与关键场景

### 7.1 用户故事

#### 场景 1：查看环境总览

作为农户，我希望打开 APP 后一次性看到温度、湿度、光照、二氧化碳和土壤湿度，以便快速判断种植环境是否正常。

#### 场景 2：执行手动灌溉

作为农户，我希望在土壤湿度过低时一键启动灌溉，并在短时间内看到控制结果，以便快速处理环境问题。

#### 场景 3：执行手动补光

作为农户，我希望在光照不足时快速开启补光或调节亮度，以便改善当前环境。

#### 场景 4：处理异常告警

作为管理员，我希望在环境指标超阈值时收到告警，并查看触发值、关联设备和处理状态，以便及时处理。

#### 场景 5：接入扩展设备

作为项目负责人，我希望系统能接入风扇等扩展设备，以便证明系统具备扩展性。

### 7.2 主演示路径

#### 主路径 A：首页看数

1. 用户打开应用
2. 首页展示网关状态、环境指标、设备摘要和告警摘要
3. 用户在 3 秒内完成环境判断

#### 主路径 B：手动灌溉控制

1. 用户进入灌溉页
2. 用户设置灌溉时长并点击开始
3. 页面进入提交中状态
4. 3 秒内返回成功、失败或处理中结果
5. 页面展示最终设备状态和操作记录

#### 主路径 C：异常告警或补光演示

1. 系统接收异常环境数据或低光照数据
2. 首页和告警页出现异常提示，或补光逻辑被触发
3. 用户看到异常结果、处理状态或规则执行结果

### 7.3 异常路径

#### 异常路径 A：网关离线

1. 应用检测网关不可达
2. 首页展示全局离线提示
3. 控制入口禁用或给出风险提示
4. 用户可进入设置页重试连接

#### 异常路径 B：设备离线

1. 目标设备状态变更为离线
2. 设备页显示离线标识
3. 控制入口禁用
4. 页面展示原因或失败提示

## 8. 关键体验路径（KEP）

### 8.1 主演示路径

#### KEP1：查看环境总览

- KEP ID：KEP1-01
- 优先级：P0
- 用户故事：用户打开 APP 后，快速查看环境与设备状态。
- 验收标准：
  - [ ] 首页首屏 3 秒内展示温度、湿度、光照、二氧化碳、土壤湿度 5 项指标
  - [ ] 指标缺失时显示 `--`，且不阻塞其他卡片显示
  - [ ] 首页展示网关在线状态、在线设备数和告警摘要

#### KEP2：手动灌溉控制

- KEP ID：KEP1-02
- 优先级：P0
- 用户故事：用户手动启动或停止灌溉设备。
- 验收标准：
  - [ ] 用户可对水泵或水阀执行启动/停止控制
  - [ ] 控制动作发起后 3 秒内出现成功、失败或处理中状态
  - [ ] 成功时页面展示最终状态，失败时展示明确失败原因或错误码

#### KEP3：手动补光控制

- KEP ID：KEP1-03
- 优先级：P0
- 用户故事：用户根据光照情况手动启动 LED 补光或切换亮度。
- 验收标准：
  - [ ] 用户可切换 LED 开关或亮度等级
  - [ ] 控制发起后 3 秒内获得明确反馈
  - [ ] 设备离线时禁止误控并展示提示

### 8.2 辅助演示路径

#### KEP4：阈值预警

- KEP ID：KEP1-04
- 优先级：P1
- 用户故事：用户设置温湿度等阈值并收到异常告警。
- 验收标准：
  - [ ] 支持至少 3 类阈值规则配置
  - [ ] 异常条件满足后 5 秒内可生成告警记录或页面提示
  - [ ] 告警详情至少包含等级、内容、触发值、时间和状态

#### KEP5：联动规则

- KEP ID：KEP1-05
- 优先级：P1
- 用户故事：用户希望系统在条件满足时自动执行控制逻辑。
- 验收标准：
  - [ ] 支持至少 2 条联动规则配置
  - [ ] 规则触发后可生成执行记录
  - [ ] 规则配置不完整时禁止保存

#### KEP6：扩展设备接入

- KEP ID：KEP1-06
- 优先级：P1
- 用户故事：系统接入一类新设备，证明平台具备扩展性。
- 验收标准：
  - [ ] 至少 1 类扩展设备完成接入
  - [ ] APP 可查看其在线状态并执行基础控制
  - [ ] 新设备接入不破坏现有设备模型和页面渲染

### 9.2 功能规格

#### FR-01 / FR-02 / FR-03 智能控制中心

- 功能描述：实现网关连接管理、环境总览、设备状态汇总和系统级状态展示
- 输入：网关地址、设备接入信息、实时环境数据
- 输出：网关在线状态、环境卡片、设备列表、异常摘要
- 约束条件：
  - 必须区分网关离线和设备离线
  - 首页渲染不依赖所有数据全部成功返回
  - 任一指标缺失时不能导致整页不可用

#### FR-04 / FR-05 智能灌溉功能

- 功能描述：采集土壤湿度相关数据，支持人工灌溉、灌溉建议和状态反馈
- 输入：土壤湿度值、灌溉阈值、用户灌溉指令、灌溉时长
- 输出：灌溉建议、设备执行状态、灌溉记录
- 约束条件：
  - 低网速场景需避免重复下发指令
  - 自动灌溉与手动灌溉冲突时，单次手动操作优先
  - 控制提交后必须出现提交中状态

#### FR-06 / FR-07 智能光照功能

- 功能描述：采集光照强度，支持 LED 补光控制、亮度调节和自动补光模式
- 输入：光照强度、补光阈值、手动亮度等级
- 输出：LED 状态、亮度等级、自动补光执行记录
- 约束条件：
  - 亮度控制应提供离散档位
  - 自动模式应支持手动关闭
  - 设备离线时禁止发送控制命令

#### FR-08 / FR-09 智能环境监测与告警

- 功能描述：实时展示温度、湿度、光照、二氧化碳等指标，并支持阈值告警
- 输入：多类环境传感器数据、阈值规则
- 输出：实时数据展示、告警记录、异常状态高亮
- 约束条件：
  - 异常状态提示必须明确易懂
  - 告警列表必须能查看等级、内容、状态和触发时间
  - 告警详情必须包含触发值和关联对象

#### FR-10 / FR-11 联动规则

- 功能描述：支持配置条件-动作规则并记录执行结果
- 输入：联动条件、动作参数、设备选择
- 输出：规则配置结果、规则执行记录
- 约束条件：
  - 规则条件不完整时禁止保存
  - 目标设备不存在或离线时，规则执行应记录失败结果
  - 同一设备同一时刻只允许一个生效动作

#### FR-12 扩展设备功能

- 功能描述：以风扇为推荐扩展设备，验证接入、监测、控制与规则联动能力
- 输入：扩展设备状态、用户控制指令、联动规则触发条件
- 输出：扩展设备状态展示、控制执行结果
- 约束条件：
  - 接入方式应复用统一设备抽象
  - 扩展设备不应破坏已有设备模型和接口结构

## 10. 非功能需求

### 10.1 性能指标

- NFR-P-01：首页首屏 3 秒内展示关键内容
- NFR-P-02：控制动作发起后 3 秒内出现成功、失败或处理中状态
- NFR-P-03：实时数据采用 5 秒级刷新或推送补齐

### 10.2 稳定性

- NFR-S-01：网关断连后系统应在 5 秒内展示异常状态
- NFR-S-02：单设备异常不应导致整页崩溃
- NFR-S-03：核心页面不应因单次接口失败而不可访问

### 10.3 安全性

- NFR-SEC-01：关键控制动作应具备明显反馈，避免误触后无感知
- NFR-SEC-02：接口保留鉴权和签名扩展位
- NFR-SEC-03：页面不直接暴露底层协议细节

### 10.4 兼容性

- NFR-C-01：目标平台为 OpenHarmony
- NFR-C-02：工程按 Stage 模型组织
- NFR-C-03：V1 默认优先适配单场景演示，不要求多温室复杂分组；多设备区域化展示仅作为可选增强项，不进入当前版本交付约束

### 10.5 可演示性

- NFR-D-01：主演示路径可连续执行一次通过
- NFR-D-02：异常场景可展示离线或告警处理结果
- NFR-D-03：扩展能力可通过新增设备进行说明和演示

## 11. 页面级需求说明

### 11.1 首页总览页

- 页面目标：让用户在 3 秒内掌握当前环境状态和设备在线情况
- 页面内容：环境指标卡片区、设备状态区、告警摘要区、快捷控制区
- 页面要求：
  - 至少展示 5 类关键指标
  - 异常指标必须高亮
  - 网关离线时必须显示全局状态提示

### 11.2 设备页

- 页面目标：统一展示接入设备，支持设备详情和手动控制
- 页面内容：设备列表、设备状态、设备详情页或抽屉、控制按钮区
- 页面要求：
  - 设备类型必须可区分
  - 离线设备必须不可误控
  - 控制中状态必须和最终状态分离展示

### 11.3 灌溉页

- 页面目标：支持查看土壤湿度和执行灌溉动作
- 页面内容：土壤湿度展示、灌溉建议、手动控制按钮、最近灌溉记录
- 页面要求：
  - 控制动作必须有清晰反馈
  - 自动与手动模式冲突时需明确说明当前生效策略

### 11.4 光照页

- 页面目标：支持查看光照强度、配置自动补光和手动补光
- 页面内容：当前光照指标、自动补光开关、亮度等级选择、补光记录
- 页面要求：
  - 手动和自动模式应可区分显示
  - 自动模式开启时需展示当前生效阈值和目标亮度

### 11.5 告警页

- 页面目标：汇总所有异常并提供状态追踪
- 页面内容：告警列表、告警等级筛选、告警详情、处理状态
- 页面要求：
  - 告警详情至少包含等级、内容、触发值、状态、关联设备和时间

### 11.6 设置页

- 页面目标：完成网关配置和规则管理
- 页面内容：网关连接配置、阈值规则配置、联动规则配置、扩展设备配置
- 页面要求：
  - 保存前必须做表单校验
  - 保存后必须展示成功或失败反馈

## 12. 风险评估

### 12.1 技术风险

- 硬件协议不稳定
- 网关接口定义变更频繁
- 自动联动逻辑边界不清

### 12.2 进度风险

- UI 开发先行但底层链路未打通
- 测试与文档滞后于开发

### 12.3 缓解措施

- 提前定义统一协议与 mock 数据
- 优先打通最小控制闭环
- 每阶段同步维护文档与测试用例
- 主路径先验收，再进入增强项开发
