# 完整文档结构与交付规划

## 1. 目标

赛题明确要求提交需求、设计、测试、数据库、测试报告、安装包等材料。因此项目需要一套完整、可评审、可复用的文档体系，而不是仅有零散说明。

## 3. 各目录职责说明

### 3.1 `doc/ref`

研究和策划层材料，服务于立项、评审、技术路线确认。

### 3.2 `doc/product`

需求层材料，回答“为什么做、做什么、做到什么程度”。

### 3.3 `doc/design`

设计层材料，回答“系统怎么做”。

### 3.4 `doc/test`

验证层材料，回答“怎么证明系统做成了”。

### 3.5 `doc/db`

数据层材料，回答“系统数据如何组织、存储、初始化”。

### 3.6 `doc/delivery`

答辩和比赛交付层材料，回答“如何高效对外展示项目成果”。

## 4. 赛题必交材料与文档映射

| 赛题要求 | 对应建议文档 |
|---|---|
| 需求分析文档 | `doc/product/PRD_智慧农业控制系统.md`、`doc/product/非功能需求清单.md` |
| 系统设计文档 | `doc/design/系统总体设计.md`、`doc/design/硬件系统设计.md`、`doc/design/软件系统概要设计.md`、`doc/design/软件系统详细设计.md` |
| 测试案例文档 | `doc/test/测试案例.md`、`doc/test/异常场景测试.md` |
| 数据库文件 | `doc/db/表结构设计.md`、初始化 SQL/数据说明 |
| 测试报告文档 | `doc/test/测试报告.md` |
| 产品 HAP 安装包 | 构建产物，不属于本次文档研究范围 |

## 5. 每份核心文档至少应包含的章节

### 5.1 产品需求文档

- 项目背景
- 目标用户
- 使用场景
- 核心功能清单
- 关键流程
- 非功能需求
- 交付边界

### 5.2 系统总体设计

- 架构目标
- 总体架构图
- 端到端流程
- 部署关系
- 模块边界

### 5.3 软件详细设计

- 模块职责
- 页面设计
- 状态流转
- 接口定义
- 异常处理
- 扩展点设计

### 5.4 测试案例文档

- 测试环境
- 功能测试
- 异常测试
- 联动测试
- 边界测试
- 预期结果

## 6. 文档产出顺序建议

不要一次性并行写所有文档，建议按依赖关系推进：

1. `ref` 研究结论
2. `product` 需求文档
3. `design` 总体设计
4. `design` 详细设计与数据库设计
5. `test` 测试方案与测试案例
6. `delivery` 展示材料

## 7. 适合比赛评审的写法建议

- 文档尽量图表化，不要只有长段文字
- 每个模块说明“输入、处理、输出、异常”
- 每个关键功能都对应一个测试项
- 每个功能都尽量体现国产化、可扩展、可控制、可验证
- 演示材料与设计文档保持口径一致

## 8. 结论

最有价值的不是“文档数量很多”，而是“文档能形成链条”：

- 需求决定设计
- 设计决定实现
- 实现接受测试
- 测试支撑答辩与交付

只要文档结构按这条链组织，项目完整度会明显高于只做页面展示的方案。
