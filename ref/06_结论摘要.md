# 结论摘要

## 1. 总结结论

针对 `doc/DOC_target.md` 的研究结论如下：

- 该题本质上是一个基于 OpenHarmony 的智慧农业“监测 + 控制 + 联动 + 扩展”应用系统。
- 要想高质量完成，不应只把它当作一个页面展示项目，而要把它按“设备层 - 网关层 - 应用层 - 展示层”四层系统来策划。
- 从开发可行性、比赛适配性、系统能力匹配度来看，最推荐采用 `Stage 模型 + UIAbility + ArkTS + ArkUI` 的原生技术路线。
- 可以使用 React，但更适合作为局部图表或 Web 子页面增强方案，不建议作为主界面路线。
- 没有必要依赖重型第三方业务框架，直接复用 OpenHarmony 官方应用框架、数据能力和测试能力即可。

## 2. 最推荐的开发方案

### 技术主线

- 应用框架：OpenHarmony Stage 模型
- 界面技术：ArkTS + ArkUI
- 应用结构：单 UIAbility + 多页面
- 数据存储：Preferences + RDB
- 通信建议：APP 到网关采用 HTTP + WebSocket
- 测试体系：Test Kit + 联调测试 + 演示场景验证

### 功能主线

- 智能控制中心
- 智能灌溉
- 智能光照
- 智能环境监测
- 1 类扩展设备接入

### 实施主线

- 先完成需求与设计文档
- 再打通最小控制闭环
- 再补齐主功能与扩展设备
- 最后集中做测试与交付材料

## 3. 对 React 的最终判断

- 可以用，但不建议作为主路线。
- 如果团队非常熟悉 React，可把它作为 Web 组件承载的局部方案。
- 对本题而言，原生 ArkTS/ArkUI 更稳妥，也更符合题意。
