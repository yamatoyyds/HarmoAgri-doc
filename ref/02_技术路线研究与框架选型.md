# 技术路线研究与框架选型

## 1. 研究目标

本文件回答以下问题：

- 基于 OpenHarmony 从零做这类应用，推荐采用什么技术路线？
- 是否存在可直接复用的现成框架？
- 是否可以使用 React 搭建界面？
- 对比赛型项目而言，哪种方案性价比最高？

## 2. 基础平台结论

结合仓库内 OpenHarmony 文档，可以确认以下基础事实：

- OpenHarmony 应用开发主线推荐基于 Stage 模型。
- 带界面的应用主体使用 `UIAbility`。
- 页面层原生推荐技术是 ArkTS + ArkUI。
- Web 相关能力由 ArkWeb / Web 组件承载。
- 数据持久化可选 Preferences、RDB、KV 等机制。
- 自动化测试由 Test Kit 提供单元、UI、性能测试能力。

## 3. 可行技术路线对比

### 路线 A：ArkTS + ArkUI 原生应用

特征：

- 使用 Stage 模型创建 OpenHarmony 应用
- 以 `UIAbility + 多页面` 方式组织前端
- 以 ArkTS 编写页面逻辑与状态管理
- 使用系统能力完成本地存储、网络访问、设备交互

优点：

- 与 OpenHarmony 官方推荐路径完全一致
- 框架适配风险最低
- 调用系统能力最直接
- 更有利于体现“OpenHarmony 原生能力”
- 演示和答辩时说服力更强

缺点：

- 团队若更熟悉 React，需要适应 ArkTS/ArkUI 语法
- 现成业务组件生态不如 Web 前端丰富

结论：最推荐。

### 路线 B：OpenHarmony 原生壳 + Web 页面（React）

特征：

- 主体仍是 OpenHarmony 应用
- 页面通过 Web 组件承载 H5/React 界面
- 原生层负责设备能力、权限、文件与本地接口
- Web 层负责业务页面和交互展示

优点：

- 前端团队可快速使用 React 构建 UI
- Web 生态成熟，图表、表单、状态管理更丰富
- 页面原型和演示效果成型更快

缺点：

- 需要额外设计 Web 与原生通信桥
- 页面性能、生命周期、资源加载、离线资源处理更复杂
- 比赛评审可能认为对 OpenHarmony 原生能力体现不够深
- 若设备控制大量依赖原生接口，桥接层会成为复杂点

结论：可行，但更适合作为“界面加速方案”，不适合作为唯一核心路线。

### 路线 C：纯 Web/React 方案再外包为容器应用

特征：

- 将应用主体视作前端站点
- OpenHarmony 仅作为运行壳

结论：不推荐。原因是赛题强调 OpenHarmony 适配、应用安装与运行、系统能力协同，纯 Web 容器化会弱化题目本身的技术价值。

## 4. 有没有现成框架可以用

### 4.1 可以直接依赖的不是“完整业务框架”，而是“平台能力框架”

当前更现实的判断是：

- 有现成的平台开发框架：Stage 模型、UIAbility、ArkUI、ArkData、ArkWeb、Test Kit。
- 没有看到仓库内直接针对“智慧农业控制系统”的现成业务框架。
- 比赛项目应优先复用平台能力和工程模板，而不是寻找过重的第三方全家桶。

### 4.2 实际可复用的框架形态

建议将“现成框架”理解为以下几类：

- 工程模板：Empty Ability / Stage 模型项目模板
- 页面框架：ArkUI 组件体系、Navigation 路由体系
- 数据框架：Preferences + RDB 的轻量组合
- 测试框架：Test Kit
- Web 补充框架：React + Web 组件（若团队熟悉 Web 技术）

### 4.3 对本项目可复用的“通用业务框架骨架”

可以自行定义一套轻量业务骨架，而不需要引入庞大框架：

- 页面壳：Dashboard、Device、Rules、Alerts、Settings
- 数据壳：API Service、Repository、Local Cache
- 领域壳：Sensor、Irrigation、Lighting、Environment、Extension Device
- 设备壳：Gateway Adapter、Protocol Adapter、Device Driver Mock

这类骨架更适合比赛型项目，便于控复杂度。

## 5. React 能不能用

结论：能用，但不建议作为主方案；建议作为备选或局部方案。

### 5.1 为什么说能用

- OpenHarmony 提供 Web 组件能力，ArkWeb 可承载网页内容。
- 这意味着 React 前端理论上可以打包为静态网页并在 Web 组件中运行。
- 对图表、表单、复杂可视化页面，React 的开发效率可能更高。

### 5.2 为什么不建议作为主方案

- 赛题重点是 OpenHarmony 原生应用，不只是把一个网页塞进壳里。
- 设备控制、网关状态、权限与系统能力调用最终仍需原生层介入。
- 双端架构会引入桥接、调试、生命周期同步、资源分发等额外成本。
- 对学生/比赛项目来说，时间往往比技术自由度更珍贵。

### 5.3 更合理的 React 使用方式

如果团队强依赖 React，建议采用“混合策略”：

- 主应用框架使用 ArkTS + ArkUI。
- 大屏统计、图表总览、复杂报表页可评估 Web/React 嵌入。
- 核心控制链路、设备管理、告警、设置页仍保持原生实现。

### 5.4 最终建议

- 首选：全 ArkTS + ArkUI。
- 备选：ArkTS 原生壳 + React 图表子页面。
- 不建议：React 统管全部页面。

## 6. 数据与通信选型建议

### 6.1 本地存储

- Preferences：用于轻量配置，例如阈值、开关、主题、最近连接设备。
- RDB：用于历史传感器数据、设备状态记录、告警日志、操作日志。

### 6.2 网络与设备通信

由于题目未明确协议，方案层建议保持可替换：

- APP 与网关：HTTP 或 WebSocket 优先
- 网关与设备：串口、MQTT、蓝牙、Wi-Fi、Modbus 等由硬件方案决定
- 比赛演示：可优先选择实现成本更低、可调试性更高的协议组合

推荐策略：

- APP 到网关：HTTP + WebSocket
- 网关到传感器/执行器：由硬件端决定，APP 仅面对统一接口

## 7. 推荐选型

### 7.1 最终推荐技术栈

- 应用模型：Stage 模型
- UI 框架：ArkTS + ArkUI
- 应用入口：单 `UIAbility` + 多页面
- 本地存储：Preferences + RDB
- 通信方式：HTTP + WebSocket
- 测试：Test Kit + 手工联调
- 图表：优先原生组件实现，复杂可视化再评估 Web 组件

### 7.2 为什么这个组合最合适

- 符合官方主线
- 适合比赛交付
- 复杂度可控
- 扩展空间足够
- 便于后续补充设备接入和测试文档

## 8. 结论

对本赛题而言，真正值得复用的是 OpenHarmony 原生应用框架，而不是额外引入一套重 Web 化框架。React 可以用，但更适合局部增强而不是全盘主导。若目标是稳妥、高完成度、可答辩、可落地，最佳路线仍然是：

- 以 ArkTS + ArkUI 为主
- 以 Stage 模型为工程基础
- 以网关抽象和设备适配层为系统扩展核心
