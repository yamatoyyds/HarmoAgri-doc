# Windows 主环境下 OpenHarmony 应用开发与调试手册

## 1. 文档目的

本文档用于在本项目中固定一条更简洁、可复现、可交付的开发路线：

- 明确以 `Windows` 作为唯一主开发环境。
- 不再将 `WSL` 纳入 OpenHarmony App 的主开发和调试方案。
- 结合本项目已有研究结论，给出从环境准备、工程搭建、联调、真机调试到交付验证的完整操作手册。

本文档适用于 `doc/DOC_target.md` 所定义的“基于 OpenHarmony 的智慧农业控制系统”项目。

## 2. 适用范围与前提

### 2.1 适用范围

- OpenHarmony 原生应用开发
- `Stage` 模型应用工程搭建
- `ArkTS + ArkUI` 页面开发
- `UIAbility` 应用入口组织
- `HAP` 构建、安装、启动和真机调试
- 与网关服务的 `HTTP + WebSocket` 联调

### 2.2 前提结论

本项目已经在其他文档中明确以下技术主线：

- 应用模型：`Stage 模型`
- 页面技术：`ArkTS + ArkUI`
- 入口模式：`单 UIAbility + 多页面`
- 数据存储：`Preferences + RDB`
- 通信方式：`HTTP + WebSocket`

## 3. 为什么采用 Windows 单主环境

### 3.1 结论

对本项目而言，推荐直接使用 `Windows` 作为唯一主环境，原因如下：

- `DevEco Studio`、`OpenHarmony SDK`、签名、设备连接、`hdc` 调试链路在 Windows 下更直接。
- 比赛项目强调稳定性、可演示性和交付效率，单环境更容易复现和排障。
- 真机安装、启动、日志查看、截图和故障定位在 Windows 下操作路径更短。
- 不引入 `WSL` 可以降低路径映射、网络转发、USB 调试、文件系统差异带来的复杂度。

### 3.2 不再考虑 WSL 的含义

本文档中的“不再考虑 WSL”包括：

- 不使用 WSL 作为 App 工程开发环境。
- 不使用 WSL 作为真机调试环境。
- 不依赖 WSL 运行接口 mock、脚本或本地服务。

如果后续确实需要本地模拟网关或辅助脚本，也统一在 Windows 中运行。

## 4. 推荐的 Windows 开发总方案

### 4.1 总体原则

- 只保留一套开发环境：`Windows + DevEco Studio + OpenHarmony SDK`
- 先完成最小闭环，再扩展功能模块
- 先用本地 mock 或静态数据打通页面与控制流程，再接真实网关
- 所有构建、安装、启动、调试、截图、交付动作都在 Windows 下完成

### 4.2 对本项目的落地形式

建议本项目按以下结构推进：

- App 本体：OpenHarmony 原生应用
- 页面主线：`Dashboard`、`Device`、`Irrigation`、`Lighting`、`Alert`、`Settings`
- 服务主线：`GatewayService`、`DeviceService`、`IrrigationService`、`LightingService`、`AlertService`、`RuleService`
- 数据主线：`Preferences` 保存配置，`RDB` 保存结构化记录
- 联调主线：Windows 本地运行 mock 网关或连接真实网关

## 5. Windows 环境准备

### 5.1 操作系统建议

- Windows 10 64 位或 Windows 11 64 位
- 建议为最新稳定补丁版本

### 5.2 必装工具

- `DevEco Studio` 4.0 及以上版本
- `OpenHarmony SDK`
- `Git`
- `Python 3.9+` 或更高版本
- `Node.js LTS`（如果本地 mock 或工具脚本使用 Node 生态）

### 5.3 建议安装内容

在 `DevEco Studio` 中至少保证以下组件可用：

- `hvigorw`
- `hdc`
- Java 运行时
- OpenHarmony 对应 API 版本 SDK
- 签名和调试相关基础组件

### 5.4 工程目录建议

建议将项目放在 Windows 本地磁盘路径，例如：

```text
D:\workspace\smart-agri-oh
```

建议：

- 路径尽量简短
- 路径中避免中文和空格
- 工程不要放在同步盘、桌面或下载目录

## 6. 真机与调试设备准备

### 6.1 设备要求

- OpenHarmony 目标设备或可用于调试的兼容设备
- USB 数据线稳定可用

### 6.2 设备端设置

在设备上完成以下操作：

- 打开开发者选项
- 打开 `USB Debugging`
- 打开 `HDC Debugging`
- 首次连接电脑时确认授权

### 6.3 PC 侧验证项

在 Windows 上需要验证：

- 设备可被 `hdc` 识别
- `DevEco Studio` 能发现目标设备
- 安装和启动链路可执行

## 7. 推荐工程结构

### 7.1 应用结构建议

本项目推荐采用：

- 单 `UIAbility`
- 多页面
- 服务分层
- 页面与设备协议解耦

### 7.2 目录结构建议

```text
entry/
├─ ets/
│  ├─ pages/
│  ├─ components/
│  ├─ services/
│  ├─ repositories/
│  ├─ domain/
│  └─ models/
└─ resources/
```

### 7.3 页面优先级

建议按以下顺序推进页面：

1. `Dashboard`
2. `Device`
3. `Irrigation`
4. `Lighting`
5. `Alert`
6. `Settings`

原因：

- 前四项最直接支撑赛题核心闭环
- 告警和设置更适合作为第二阶段收口

## 8. 本项目开发顺序

### 8.1 阶段 0：环境验证

目标：确认 Windows 工具链完整可用。

完成项：

- `DevEco Studio` 可正常启动
- `OpenHarmony SDK` 安装完成
- 设备可连接
- 能识别 `hdc`、`hvigorw`、Java 相关工具

### 8.2 阶段 1：工程骨架搭建

目标：建立 OpenHarmony Stage 工程与基础页面结构。

完成项：

- 建立 `Stage` 模型工程
- 建立单 `UIAbility`
- 建立基础路由和页面目录
- 搭好 `services`、`domain`、`models` 等目录结构

### 8.3 阶段 2：最小闭环打通

目标：优先完成“看数据、控设备、回状态”的最小演示链路。

建议闭环：

- 首页展示环境数据
- 设备页展示设备状态
- 至少一个执行器可控制
- 控制结果可回显

### 8.4 阶段 3：核心模块扩展

目标：完成赛题主功能。

重点事项：

- 灌溉模块
- 光照模块
- 环境监测模块
- 告警模块
- 联动规则模块

### 8.5 阶段 4：联调与稳定性增强

目标：完成网关联调和异常场景处理。

重点事项：

- 接真实网关或 Windows 本地 mock 服务
- 处理离线、失败、超时、重试场景
- 补齐历史记录与状态提示

### 8.6 阶段 5：测试与交付

目标：形成可安装、可运行、可演示、可提交的最终版本。

完成项：

- 功能验证
- 异常验证
- 联调回归
- `HAP` 产物输出
- 测试记录与演示资料整理

## 9. 本地调试与联调策略

### 9.1 本地 mock 策略

由于硬件与网关可能不会第一时间完全就绪，建议统一采用以下策略：

- 第一阶段：静态数据驱动页面和交互
- 第二阶段：Windows 本地运行 mock 服务
- 第三阶段：切换为真实网关接口

### 9.2 通信策略

与项目设计文档保持一致：

- `HTTP`：首屏查询、配置提交、手动控制
- `WebSocket`：实时状态推送、在线状态变化、告警推送

### 9.3 联调顺序

建议固定为：

1. 首页环境数据查询
2. 设备列表查询
3. 单个设备控制
4. 控制结果回显
5. WebSocket 状态推送
6. 告警与联动验证

## 10. 构建、安装、启动与调试闭环

### 10.1 推荐操作顺序

每次需要验证版本时，按以下顺序执行：

1. 环境检查
2. 构建应用
3. 安装应用
4. 启动应用
5. 验证页面与主流程
6. 必要时截图和导出日志

### 10.2 推荐命令链路

结合项目中的 `ohos-app-build-debug` 参考能力，可使用如下命令思路：

```bash
python3 <skill-dir>/scripts/env_detector.py
python3 <skill-dir>/scripts/build.py --dir <project-dir> --show-env
python3 <skill-dir>/scripts/install.py -f <hap-file>
python3 <skill-dir>/scripts/launch.py --dir <project-dir>
python3 <skill-dir>/scripts/screenshot.py -o ./screenshots
```

说明：

- 实际执行时需将 `<skill-dir>` 替换为技能目录路径
- `<project-dir>` 替换为项目工程路径
- `<hap-file>` 替换为构建输出的 HAP 文件路径

### 10.3 调试时的重点观察项

- 首页是否能正确展示实时数据
- 设备控制是否能收到反馈
- 离线状态是否有清晰提示
- 页面跳转、路由和返回逻辑是否稳定
- 关键异常是否有日志和用户提示

## 11. 数据与配置落地建议

### 11.1 Preferences 适用范围

适合保存：

- 网关地址
- 自动灌溉开关
- 自动补光开关
- 阈值设置
- 最近连接信息

### 11.2 RDB 适用范围

适合保存：

- 设备信息
- 传感器历史记录
- 告警记录
- 控制日志
- 联动规则记录

### 11.3 拆分原则

- 轻量配置放 `Preferences`
- 结构化、可查询、可展示的记录放 `RDB`

## 12. 测试与验收建议

### 12.1 测试重点

参考 `doc/test/测试方案.md`，Windows 主环境下至少覆盖：

- 页面展示正确性
- 设备控制正确性
- 状态反馈正确性
- 告警触发正确性
- 联动执行正确性
- 异常恢复能力

### 12.2 重点异常场景

- 网关离线
- 设备离线
- 数据缺失
- 控制失败
- 规则冲突

### 12.3 验收最低标准

- 首页能展示关键环境数据
- 至少一个执行器可控
- 控制结果可回显
- 至少一条告警规则有效
- 至少一个扩展设备接入方案明确
- 可导出可安装的 `HAP`

## 13. 常见风险与应对

### 13.1 风险 1：环境安装不完整

现象：

- `DevEco Studio` 可打开，但 SDK 不完整
- 工具链缺少 `hdc` 或构建组件

应对：

- 在 IDE 中检查 SDK 安装状态
- 统一记录环境版本，避免团队成员版本不一致

### 13.2 风险 2：真机连接不稳定

现象：

- 设备无法识别
- 安装失败
- 调试会话中断

应对：

- 固定调试线缆和端口
- 提前完成设备授权
- 在主流程联调前先完成环境冒烟测试

### 13.3 风险 3：页面已完成但控制链路未打通

现象：

- 页面能展示，但无法形成设备控制闭环

应对：

- 优先完成一个执行器的最小控制链路
- 不把界面美化优先级放在控制链路之前

### 13.4 风险 4：真实硬件或真实网关未按时到位

现象：

- 无法按计划进行真实联调

应对：

- 在 Windows 本地保留 mock 服务方案
- 保留静态数据模式作为演示兜底

### 13.5 风险 5：技术路线摇摆

现象：

- 开发中途转向 Web 主导或其他复杂方案

应对：

- 主路线坚持 `ArkTS + ArkUI`
- React 或 Web 仅在确有必要时作为局部补充

## 14. 推荐执行清单

建议团队按以下顺序执行：

1. 在 Windows 上完成 `DevEco Studio` 和 SDK 安装
2. 验证设备连接与调试能力
3. 创建 `Stage + UIAbility + ArkTS + ArkUI` 工程
4. 建立 `pages/components/services/domain/models` 目录结构
5. 完成首页与设备页骨架
6. 打通单个设备控制闭环
7. 接入本地 mock 或真实网关
8. 补齐灌溉、光照、告警、设置模块
9. 做异常测试与回归测试
10. 输出 `HAP`、测试记录和演示材料

## 16. 结论

对于本项目，最推荐的实践不是“Windows + WSL 双环境”，而是直接收敛到 `Windows 单主环境`：

- 方案更稳定
- 调试链路更短
- 团队协作更容易统一
- 更符合比赛项目对效率、交付和演示稳定性的要求

在此基础上，继续坚持项目既定技术主线：

- `Stage 模型`
- `UIAbility`
- `ArkTS + ArkUI`
- `Preferences + RDB`
- `HTTP + WebSocket`

只要先打通“数据展示 - 设备控制 - 状态反馈”最小闭环，再逐步扩展灌溉、光照、告警和联动能力，就能形成一条可实施、可验证、可答辩的 OpenHarmony 应用落地路径。
