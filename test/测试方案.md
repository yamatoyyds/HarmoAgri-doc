# 测试方案

## 1. 测试目标

验证系统是否满足以下要求：

- P0 功能和主演示路径稳定可用
- 异常场景可感知、可恢复、可解释
- 接口契约、页面行为和测试口径一致
- 文档与实现具备追踪关系

## 2. 测试范围

### 2.1 MVP 功能测试

- 首页总览
- 网关状态显示
- 设备列表
- 手动灌溉控制
- 手动补光控制
- 阈值告警
- 扩展设备接入

### 2.2 ENHANCED 功能测试

- 自动补光
- 联动规则
- 规则执行记录

### 2.3 OPTIONAL 功能测试

- 历史记录展示
- 多设备区域化展示

### 2.4 非功能测试

- 首页性能冒烟测试
- 控制反馈时延测试
- 网关离线恢复测试
- 演示主链路一次通过验证

### 2.5 异常测试

- 网关离线
- 设备离线
- 数据缺失
- 控制失败
- 规则保存失败

## 3. 测试环境

- OpenHarmony 目标设备
- 智能主机网关或模拟网关
- 传感器/执行器或模拟设备
- 稳定网络环境
- mock 数据和真实数据两套方案

## 4. 测试策略

- 主路径优先：优先验证首页看数、手动灌溉、手动补光/告警展示
- 闭环验证：从数据展示到控制反馈完整验证
- 联动验证：条件触发到动作执行完整验证
- 回归验证：每轮修改后执行关键路径验证
- 降级验证：在离线、缺失和失败场景下验证页面降级策略

## 5. 测试类型与方法

### 5.1 功能测试

- 页面展示正确性
- 控制命令正确性
- 规则配置正确性
- 状态反馈正确性

### 5.2 接口联调测试

- HTTP 查询接口
- HTTP 控制接口
- WebSocket 推送接口
- 错误响应结构一致性

### 5.3 数据一致性测试

- 页面展示与接口返回一致
- 历史记录与控制结果一致
- 告警记录与规则触发条件一致

### 5.4 异常恢复测试

- 网关重连
- 设备恢复在线
- 页面从 `offline` 恢复到 `ready`

## 6. 测试优先级策略

- P0 用例：必须全部执行，必须通过
- P1 用例：必须执行，允许少量已知风险但不可阻塞主路径演示
- P2 用例：根据时间执行，作为增强验证，不阻塞当前版本交付

## 7. 主演示链路专项验证

### 7.1 主演示路径

1. 首页看数
2. 手动灌溉控制
3. 手动补光或告警展示

### 7.2 验证要求

- 连续执行一次通过
- 中间不依赖手工修复数据
- 失败场景下有可演示的兜底说明

## 8. 测试数据准备

### 8.1 正常数据

- 温度：24-28
- 湿度：55-70
- 光照：300-600
- 二氧化碳：500-900
- 土壤湿度：35-50

### 8.2 异常数据

- 温度：>30
- 湿度：<30 或 >85
- 光照：<200
- 二氧化碳：>1200
- 土壤湿度：<35

## 9. Mock 与兜底策略

- 若真实硬件未就绪，优先使用 mock 网关保障首页看数和控制链路演示
- 若 WebSocket 不稳定，可用 HTTP 轮询作为联调兜底
- 若扩展设备不可用，可保留已注册模拟设备进行展示说明

## 10. 进入与退出准则

### 10.1 进入准则

- PRD、接口设计、页面规格和测试案例已冻结当前版本
- 网关接口至少完成 mock 或真实接入之一
- 首页、灌溉页和光照页已可运行

### 10.2 退出准则

- P0 用例全部通过
- TC-19 主演示链路一次通过
- 关键异常场景有明确处理结果
- 测试报告可支撑交付结论

## 11. 缺陷处理规则

| 缺陷级别 | 定义 | 处理要求 |
|---|---|---|
| S1 | 阻塞演示或主流程不可用 | 必须修复 |
| S2 | 核心功能异常但有替代路径 | 优先修复 |
| S3 | 一般问题，不影响主流程 | 记录并评估 |
| S4 | 展示或文案问题 | 视时间修复 |
